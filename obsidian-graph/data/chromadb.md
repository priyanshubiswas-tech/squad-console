# ChromaDB

**Status**: built and populated — 15 collections, real content. Not yet queried by anything (that's [[langgraph-agent]]'s `rag_retriever` node, not built yet).

The vector store for tactical knowledge. Populated by [[embedding-job]] reading `knowledge_base/`, will be queried by [[langgraph-agent]]'s `rag_retriever` node.

## Collections (all populated)
- `{team}_full` — embeds `knowledge_base/{team}/tactics_notes.md`, real researched content per team. Only ever queried when that team is the *active* team.
- `{team}_public` — embeds `knowledge_base/{team}/public_scouting.md`. Queried when that team is the *opponent*.
- `shared_theory` — embeds `knowledge_base/shared/tactical_theory.md`. Always queryable.

Every chunk carries `{team, doc_type, visibility}` metadata — [[access-control]]'s `access_filter` node will re-check this metadata as a defense-in-depth layer once built, even though the collection split should already prevent leaks.

Embeddings are generated by Chroma's default local `all-MiniLM-L6-v2` sentence-transformers model - no LLM API call, no key needed for this phase.

## Persistence
Named Docker volume `chroma_data`, container `chroma` (`chromadb/chroma:0.5.20`), published to host on `8001` (container port `8000`) so [[embedding-job]] (host-run) can reach it. `knowledge_base/` itself is read directly off the host filesystem, so tactical notes can be edited live and re-embedded without a rebuild.
